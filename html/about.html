Inspired by pirate radio and the free culture movment, PirateBox is a self-contained mobile collaboration and file sharing device. PirateBox utilizes Free, Libre and Open Source software (FLOSS) to create mobile wireless file sharing networks where users can anonymously share images, video, audio, documents, and other digital content.

PirateBox is designed to be safe and secure. No logins are required and no user data is logged. The system is purposely not connected to the Internet in order to prevent tracking and preserve user privacy[*].

PirateBox was created by David Darts and is registered under a Free Art License (2011). To learn more about the project or to find out how to build your own PirateBox system, please visit http://wiki.daviddarts.com/PirateBox

SteamBox is one of the attempts to port PirateBox to Android. See https://github.com/thedod/SteamBox.

[*] Internet connectivity (and other threats):

As opposed to David Dart's standalone PirateBox, the admin of an Android based PirateBox (i.e. the phone's owner) may decide to use it as a file-exchange utility on an existing wifi network (work, friend's home, etc.).

Even when the phone is used as a portable hotspot, it may still be connected to cellular internet (and would most probably provide it to the users). This configuration is useful in rural areas, trains, etc. and IMHO - the question whether we're online or not is a minor one compared to other threats:

  • A dishonest admin can add code to PirateBox that does log activity. Even a nice person with a droid can actually be an RIAA/MPAA/MAFIAA bounty hunter for all you know.
  • People around you can sniff your activity. Especially at geek cons ;)

The first problem can only be minimized by avoiding dubious admins (use your street wisdom).

SSL can solve the second problem, as long as you check 2 things when your browser warns you about the SSL certificate:

  1. If the certificate's common name is fun2code, it's the default key that comes with the distribution. Everyone can obtain it, so using it as an SSL key is useles.
    In such a case you should even ask the admin to switch to http and avoid creating a dangerous false sense of security.
  2. To avoid man-in-the-middle attacks, you should verify the key's fingerprints with the admin, but you shouldn't do this via the network: The admin can show you the fingerprints on his/her phone's screen, hand them to you on a printed paper, etc.
